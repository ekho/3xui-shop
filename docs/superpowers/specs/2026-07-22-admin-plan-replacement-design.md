# Принудительная смена тарифа в User Editor

## Цель

Дать администратору возможность заменить действующий тариф клиента из
`Admin Tools → Пользователи`, не меняя ссылку подписки (`subId`) или VPN-учётные
данные (`vpn_id`). Сценарий нужен для ручной корректировки доступа без платежа.

## Границы

- Операция доступна только пользователям с базовым доступом `regular`; `banned`
  является накладываемым ограничением и не исключает операцию.
- Для участника `unlimited` кнопка не показывается. Безлимит назначается и снимается
  только существующим редактором групп: снятие `unlimited` уже переводит клиента на
  стартовый триал.
- Смена тарифа не переводит пользователя между `regular` и `unlimited`.
- Операция применима только к существующему клиенту панели. Никаких новых клиентов,
  серверов, платежей или миграций БД она не создаёт.

## Интерфейс и сценарий

1. В карточке подходящего пользователя появляется кнопка «Сменить тариф».
2. Бот показывает выбор, где первой строкой идёт «Триал», а далее — базовые тарифы:
   видимые планы из `PlanService.get_purchasable_plans()` без группы `unlimited`.
   Скрытый безлимитный тариф сюда не входит.
3. При выборе базового тарифа администратор выбирает срок из того же списка, который
   использует обычная покупка. При выборе «Триал» срок не выбирается: используются
   `TRIAL_PERIOD`, `BONUS_DEVICES_COUNT` и `TRIAL_TRAFFIC_GB` из конфигурации.
4. Экран подтверждения показывает пользователя, выбранный тариф/триал, срок и
   предупреждение о начале нового периода и сбросе трафика.
5. Подтверждение потребляет FSM-состояние до сетевого вызова. Повторный тап либо
   устаревший экран не запускает операцию второй раз.
6. После успеха бот перерисовывает карточку и best-effort уведомляет пользователя на
   его языке. В уведомлении нет ссылки подписки, потому что она не меняется.

## Применение на панели

Перед подтверждением обработчик заново получает пользователя и проверяет, что у него
по-прежнему нет группы `unlimited` и клиент доступен на назначенном сервере.

Для базового тарифа используется существующий `VPNService.change_subscription()` с
`devices`, `duration` и `traffic_gb` выбранного плана. Он уже соответствует поведению
обычной смены после оплаты: заменяет лимит устройств и трафика, считает `expiryTime`
от текущего момента, приводит набор инбаундов к тарифу, сбрасывает трафик и повторно
накладывает бан, если он есть.

Для «Триала» вводится небольшой явный сервисный путь принудительного сброса на
триал. Он повторно использует логику существующего снятия `unlimited`: применяет
параметры триала, набор групп `regular` (плюс `banned`, когда он задан), начинает
срок с текущего момента, сбрасывает счётчик и сохраняет бан. Он не использует гейт
`SubscriptionService.gift_trial()`: администратор может назначить триал независимо
от `TRIAL_ENABLED` и предыдущего `is_trial_used`. Флаг `is_trial_used` не сбрасывается
и не служит гейтом для этой принудительной операции.

`sub_id`, `vpn_id`, сервер клиента и его ссылка подписки остаются неизменными в обоих
вариантах.

## Ошибки и аудит

- Если пользователь стал `unlimited`, клиент исчез с панели, сервер недоступен,
  список тарифов/сроков пуст или вызов панели неуспешен, бот показывает явную ошибку
  и не сообщает об успехе.
- Ошибка после частичного ответа панели логируется; карточка перерисовывается только
  после подтверждённого успеха вызова.
- В аудит добавляется отдельное действие принудительной смены тарифа с режимом
  (`plan` или `trial`), длительностью, лимитом устройств и трафика. В зеркале аудита
  публикуется краткая безопасная сводка.

## Проверка

- Тесты клавиатур: кнопка есть только для `regular`, список не содержит скрытый
  безлимитный план и содержит «Триал».
- Тесты FSM: выбор плана → срока → подтверждение; «Триал» пропускает срок; повторный
  тап и изменение группы приводят к безопасному отказу.
- Тесты сервисов: базовый тариф передаёт параметры в `change_subscription`; триал
  применяет параметры конфигурации, не меняет идентификаторы и сохраняет бан.
- Тесты аудита и локализованного уведомления; регрессия существующих тестов.
