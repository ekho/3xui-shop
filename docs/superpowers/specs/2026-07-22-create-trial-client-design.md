# Создание клиента с триалом через админ-бота

## Цель

Дать администратору возможность выдать VPN-клиенту обычный триал, когда клиент не
может самостоятельно открыть Telegram-бота. Администратор вводит Telegram ID и
отображаемое имя, а затем передаёт клиенту созданную ссылку подписки.

## Выбранный путь

В разделе «Пользователи» появится действие «Создать клиента». Оно ведёт в
последовательный FSM-флоу: Telegram ID → имя → явное подтверждение. Этот путь
согласуется с существующими админскими операциями (начисление дней, бан и сброс
трафика), не требует запоминать команды и защищает от случайной выдачи доступа.

Командный интерфейс и создание только в панели 3x-ui не входят в объём работы.

## Поток и данные

1. Администратор открывает «Пользователи» и выбирает «Создать клиента».
2. Бот принимает положительный Telegram ID, затем имя после `strip()` длиной от 1
   до 32 символов, и показывает оба значения в экране подтверждения.
3. При подтверждении бот проверяет, что такого ID ещё нет, триал включён и в пуле
   есть сервер. Просроченное или повторное подтверждение не выполняет операцию.
4. Сервис создаёт запись `User` с новым `vpn_id` и `sub_id`, отображаемым именем,
   русской локалью по умолчанию и статусом `APPROVED`. Явно выданный
   администратором доступ не должен остаться в `PENDING`, если включён approval
   gate.
5. Тот же сервис вызывает существующий `SubscriptionService.gift_trial`. Поэтому
   используются текущие `TRIAL_PERIOD`, `TRIAL_TRAFFIC_GB` и
   `BONUS_DEVICES_COUNT`, а созданный клиент проходит стандартное назначение
   сервера, групп и панели 3x-ui.
6. После успеха бот записывает отдельное событие аудита и открывает карточку
   клиента. Карточка содержит кнопку или сообщение со ссылкой подписки для
   безопасной передачи администратором клиенту.

## Ошибки и идемпотентность

- Невалидный ID и имя не переводят FSM к следующему шагу.
- Уже существующий ID не изменяется и не получает второй триал.
- Выключенный триал, отсутствие свободного сервера, недоступность панели и пустой
  набор инбаундов приводят к понятной ошибке без успешного ответа и без аудита
  выдачи.
- Сервис компенсирует неуспешный первичный provisioning: временная запись нового
  пользователя удаляется только если панель не выдала доступ. Существующий
  пользователь никогда не удаляется этим потоком.
- Подтверждение очищает FSM до сетевого вызова; второй тап получает сообщение о
  протухшем флоу и не создаёт второй клиент.

## Компоненты

- `admin_tools/keyboard.py` и `navigation.py`: кнопки и callback-значения.
- `admin_tools/user_handler.py`: FSM, валидация, подтверждение и отображение
  результата.
- Новый узкий метод в `SubscriptionService`: атомарно оркестрирует создание
  админского пользователя и обычную выдачу триала; не дублирует provisioning из
  `VPNService`.
- `User`: безопасное удаление только что созданной записи для компенсации ошибки.
- `AuditAction` и `AuditService`: событие `user.create_trial` с ID, именем и
  параметрами триала; оно видно в истории карточки и в зеркале аудита.
- Локализации `ru` и `en`: все новые подписи, подсказки и ошибки.

## Проверка

Тесты на `unittest.IsolatedAsyncioTestCase` покрывают: успешное создание с
параметрами из конфигурации; дубликат ID; ошибку provisioning с компенсационным
удалением; а также валидацию ID/имени и одноразовое подтверждение в роутере.
Полный набор запускается стандартным `python -m unittest discover`; дополнительно
проверяются `poetry check`, компиляция исходников и извлечение/компиляция
локализаций.
