# План: дублирование и пробелы функционала 3xui-shop относительно встроенного бота 3x-ui

_Дата: 5 июля 2026. Источники: исходники встроенного Telegram-бота 3x-ui (панель v3.4.2 — целевая версия проекта, зафиксированная в памяти сессии) и полный аудит кода `bot/app/` (форк 3xui-shop)._

---

## 0. Методология

Изучены:
- **Встроенный бот 3x-ui**: исходники `internal/web/service/tgbot/{tgbot,tgbot_client,tgbot_event,tgbot_inbound,tgbot_report,tgbot_router}.go`, `internal/eventbus/events.go`, `internal/web/service/setting.go`, `frontend/src/pages/settings/TelegramTab.tsx`, локализация `en-US.json` — **зафиксировано отдельно для тега v3.4.2** (не только `main`), чтобы не полагаться на более новую архитектуру (в `main` бот уже переписан на event-bus и добавлена мульти-нодовая логика, которой в v3.4.2 в таком виде могло не быть; проверкой подтверждено, что событийная система, включая `node.up/down`, уже присутствует в v3.4.2).
- **3xui-shop** (`bot/app/`): все роутеры (`admin_tools/*`, `subscription/*`, `profile`, `referral`, `support`, `download`, `main_menu`, `misc`), все сервисы (`vpn.py`, `server_pool.py`, `subscription.py`, `referral.py`, `notification.py`, `payment_stats.py`, `invite_stats.py`, `plan.py`), все платёжные шлюзы, `tasks/*`, `middlewares/*`, `config.py`, `db/models/*`, плюс `README.md` (включая явный роадмап "🚧 Current Tasks" и зачёркнутые пункты в Overview — это фактически roadmap-заметки самого оригинального автора о том, что не доделано или было решено не делать).

## 1. Встроенный бот 3x-ui — сводка (для справки)

Это **исключительно admin-ops/notification-инструмент панели**, не бот продаж. Подтверждено: grep по `payment|invoice|checkout|price|balance|wallet` по всем файлам `tgbot_*.go` — ноль совпадений. README панели прямо пишет: *"Telegram bot for remote monitoring and management"*.

| Категория | Что умеет |
|---|---|
| Admin-команды | `/start`, `/help`, `/status`, `/id`, `/usage [email]`, `/inbound [remark]`, `/restart` (перезапуск **Xray Core**, не панели), `/clearall` |
| Управление клиентом (по email) | refresh, resetTraffic, limitTraffic, resetExpire, ipLog, ipLimit/clearIps, toggle enable/disable, привязка Telegram ID клиента (`setTGUser`/`tg_user`) |
| Self-service клиента | Если админ привязал `tgId` клиента — клиент может написать боту `/usage` и получить свой трафик/срок/статус. Привязка **только через админа**, self-service linking (код/кнопка для самостоятельной привязки) отсутствует |
| Отчёты | Server usage (CPU/RAM/disk/uptime/xray-статус) по крону (`tgRunTime`, `@daily` по умолчанию), список инбаундов, "скоро закончится" отчёт, бэкап БД панели по расписанию или по кнопке |
| Уведомления (event bus) | `login.attempt` (успех/провал входа в веб-панель), `cpu.high`, `memory.high` (пороги `tgCpu`/`tgMemory`), `xray.crash`, `outbound.up/down`, `node.up/down` (мульти-нода) |
| Пороговые алерты клиенту/админу | `expireDiff` (дней до истечения), `trafficDiff` (ГБ до лимита) — шлются и админу (сводка), и напрямую привязанному клиенту |
| Конфигурация | `tgBotEnable`, `tgBotToken`, `tgBotChatId` (админ-чаты), `tgBotProxy`, `tgBotAPIServer`, `tgRunTime`, `tgBotBackup`, `tgCpu`, `tgMemory`, `tgLang`, `tgEnabledEvents` |
| **Чего нет** | Тарифов, оплаты, инвойсов, продажи подписок, промокодов, рефералки, мульти-сервер пула с балансировкой — вообще ничего из биллинг-домена |

## 2. Дублирующий функционал → к удалению

**Итог аудита: в уже реализованном коде 3xui-shop прямых дублей встроенного бота не найдено.** Домены не пересекаются: 3xui-shop управляет *своими* сущностями (подписка/оплата/промокод/реферал), встроенный бот — сырым состоянием VPN-клиента и инфраструктурой панели. Однако есть точки, которые выглядят похоже и заслуживают явной пометки, чтобы не задублировать их **при доработке** (см. риски ниже) — специально помечаю как "не удалять, но не путать/не пересекать":

| # | Пара функций | Почему это НЕ дубль | Действие |
|---|---|---|---|
| 2.1 | `admin_tools/restart_handler.py` (`RESTART_BOT`, [restart_handler.py:18-31](bot/app/bot/routers/admin_tools/restart_handler.py)) vs встроенный `/restart` (перезапуск Xray Core) | Перезапускает **процесс Python-бота** (`os.execv`), а не Xray/панель — целевой объект другой | Оставить. Переименовать кнопку/подпись в UI на "Restart shop bot process", чтобы админ не путал с перезапуском Xray у встроенного бота |
| 2.2 | `admin_tools/backup_handler.py` (`CREATE_BACKUP`) vs встроенный `tgBotBackup`/`get_backup` | Бэкапит **свою** SQLite (`bot_database.db` — подписчики/платежи/промокоды/рефералы), а не БД панели (`x-ui.db` — инбаунды/клиенты) | Оставить оба — они бэкапят разные базы данных. Задокументировать в README/DEPLOYMENT.md, что нужны **обе** копии для полного восстановления |
| 2.3 | `admin_tools/server_handler.py` (`PING_SERVER`) vs встроенный CPU/RAM отчёт | Проверяет сетевую доступность хоста *с точки зрения бота* (для роутинга новых клиентов по пулу серверов) — у встроенного бота нет понятия "пул серверов", он отчитывается о ресурсах **своего** единственного хоста | Оставить — разная смысловая нагрузка (доступность для балансировки vs. загрузка CPU/RAM) |

### Риск на будущее (не код сейчас, а требование к дизайну)
Когда будет реализовываться **User Editor** (см. §3.2) — не переизобретать "raw" операции над VPN-клиентом (reset traffic / toggle enable-disable / изменение IP-limit / IP-log), которые встроенный бот **уже бесплатно** даёт админу через `/usage <email>` + инлайн-кнопки (`email` в 3x-ui как раз равен `str(tg_id)` в этом форке — см. [vpn.py:192](bot/app/bot/services/vpn.py)). User Editor должен ограничиться **доменом магазина** (см. §3.2), а не дублировать управление самим VPN-клиентом.

## Итог

- **Дублирования почти нет** — 3xui-shop и встроенный бот 3x-ui живут в разных доменах (биллинг vs. ops), и это архитектурно правильно. Единственное, что нужно — не наступить на грабли при реализации User Editor (§2, риск).
- **6+ пунктов backlog** (§3) с готовой обосновкой и оценкой сложности — Statistics и авто-detect online (§3.1, §3.6) — низкая цена/высокая ценность, начать с них.
- **2 стаба к удалению** (§4) — TEST-кнопка и денежные реферальные награды — обе низкой ценности при текущем состоянии кода, рекомендую вырезать, чтобы не тащить мёртвый код дальше.
