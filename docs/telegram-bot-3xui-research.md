# Ресёрч: Telegram-бот для управления 3x-ui + биллинг и оплата

_Дата: 2 июля 2026. Задача: выбрать оптимальный способ сделать бота для управления панелью (в первую очередь клиентами) с биллингом и приёмом оплаты, вместо громоздкого решения на n8n._

---

## TL;DR / краткий вывод

1. **n8n — тупик для этой задачи.** Твоя боль реальна: в Telegram-ноде n8n нет нормальной поддержки платёжного флоу (`sendInvoice` → `pre_checkout_query` → `successful_payment`), и стейт биллинга в workflow-движке вести неудобно. n8n стоит оставить максимум под нотификации/крон-джобы, но не под сам бот.

2. **Главная развилка — не «какой бот», а «какая панель».** Экосистема реселлинг-ботов массово мигрировала с 3x-ui на **Remnawave**. Флагманский бот-магазин под 3x-ui (`snoups/3xui-shop`, 443★) **заархивирован 2 ноября 2025** и переехал на Remnawave (`remnashop`). Если биллинг — это всерьёз и надолго, стоит рассмотреть переход на Remnawave.

3. **Не пиши с нуля, если можно форкнуть.** Под обе панели есть зрелые open-source магазины (MIT), где уже готовы Stars, YooKassa, крипта, триалы, рефералка, промокоды, админка и мульти-сервер. Своя разработка (aiogram 3 + py3xui) оправдана, только если нужны нестандартные сценарии.

4. **Платежи: комбо, а не один провайдер.** Рабочая связка для RU/СНГ — **YooKassa (карты/СБП) + CryptoBot или Cryptomus (крипта) + Telegram Stars (iOS/анонимные)**. Stars нельзя делать основным рельсом: эффективная комиссия ~32% на мобиле.

## 1. Главная развилка: остаться на 3x-ui или перейти на Remnawave

У тебя уже развёрнут 3x-ui, поэтому это ключевое стратегическое решение до выбора бота.

| Критерий | **3x-ui** (MHSanaei) | **Remnawave** |
|---|---|---|
| Архитектура | Монолит на Go: панель + Xray-core в одном процессе | Микросервисы: панель (NestJS+React) отдельно от нод (Remnanode) |
| БД | SQLite | PostgreSQL (транзакции, журналирование, устойчивость к падениям) |
| Мульти-сервер | Второй сервер = отдельная копия панели **без синхронизации** | Неограниченно нод, один юзер доступен везде, панель рулит нодами через REST |
| API | Базовый | Полноценный REST + официальные SDK (Python/Go/TypeScript) |
| Известная беда | SQLite бьётся при жёстких перезагрузках («file is not a database»), сбрасывается статистика | — (новее, но экосистема быстро растёт) |
| Инструмент миграции | — | `remnawave/migrate` — перенос юзеров с 3x-ui и др. панелей |
| Кому подходит | Личное использование, 1 сервер, минимум сложности | Сервис с несколькими локациями, автоматизация, надёжность данных, реселлинг |

**Вывод по развилке.** Если это растущий бизнес с планами на несколько локаций и стабильный биллинг — **Remnawave** объективно лучше приспособлен под реселлинг, и лучшие боты сейчас развиваются именно под него. Если пока это небольшой/личный сценарий на одном сервере и не хочется переезда — 3x-ui рабочий вариант, зрелый SDK (`py3xui`) и живые боты есть. Мигрировать позже можно официальным `remnawave/migrate`.

## 2. Три пути реализации

### Путь A. Доработать n8n — **не рекомендую**
Telegram-нода n8n не умеет платёжный протокол Telegram (инвойсы, pre-checkout, вебхук успешной оплаты) — всё приходится собирать вручную на HTTP Request + отдельном вебхуке, плюс стейт клиента/подписки в workflow вести неудобно. Отсюда и громоздкость. n8n оставь под то, что он делает хорошо: крон-нотификации, интеграции, «клеевые» задачи.

### Путь B. Форкнуть готовый магазин — **быстрее всего**
Под 3x-ui и Remnawave есть готовые боты с оплатой, триалами, рефералкой, промокодами и админкой. Ставятся через Docker за вечер, конфиг — через `.env`. Дальше кастомизируешь под себя. Подходит, если сценарии продаж стандартные.

### Путь C. Свой бот на aiogram 3 + py3xui — **максимум контроля**
Оправдано, если нужны нестандартные тарифы/логика, глубокая интеграция с внутренними системами или ты хочешь полностью владеть кодом. Больше работы, но полная гибкость. Ниже — стек.

## 4. Стек для своего бота (Путь C)

- **Фреймворк: `aiogram 3` (Python, async).** Де-факто стандарт, first-class поддержка Telegram Payments: `sendInvoice`, `pre_checkout_query`, `successful_payment`, и уже есть поля подписок/рекуррента (`subscription_expiration_date`, `is_recurring`, `is_first_recurring`). Именно то, чего не хватало в n8n. Альтернативы на Node — Telegraf/grammY, но под 3x-ui экосистема Python богаче.
- **API-обёртка панели: `py3xui` (iwatkot).** Зрелый sync+async SDK для 3x-ui, активный (релиз в июне 2026), на PyPI. Для управления несколькими панелями — `py-multi-3xui`. На Go есть `client3xui`. Под Remnawave — официальные SDK.
- **БД: PostgreSQL.** Клиенты, подписки, платежи, промокоды, рефералы, история.
- **Планировщик: APScheduler / cron.** Проверка истечения подписок, напоминания, авто-отключение по окончании, авто-продление.
- **Деплой: Docker Compose** (бот + БД + reverse proxy с TLS под вебхуки платёжек).

Ориентир по объёму: базовый рабочий бот с одним платёжным провайдером — это дни, а не недели. Именно поэтому форк готового (Путь B) часто выгоднее старта с нуля.

## 5. Платежи и биллинг (твоя основная боль)

### Сравнение рельсов

| Провайдер | Что это | Комиссия | Юр. лицо | Рекуррент | Когда брать |
|---|---|---|---|---|---|
| **Telegram Stars (XTR)** | Нативная валюта Telegram | **~32% на мобиле** (30% Apple/Google + спред Fragment), ~3–4% на десктопе | Не нужно | Да (подписки Stars) | iOS-аудитория, анонимные покупатели. **Не основной рельс** из-за комиссии |
| **YooKassa** | RU-эквайринг | ~3.5% | ИП/ООО | Да — только через API | Основной рельс для RU-карт, СБП, SberPay, чеки по 54-ФЗ |
| **YooMoney** | Кошелёк | выше/проще | Физлицо | Ограниченно | Быстрый старт без юрлица, меньше доверия |
| **CryptoBot / Crypto Pay** | Крипто-провайдер внутри Telegram (от команды, близкой к TG) | низкая (~0–3%) | Не нужно | — | Крипта в 2 тапа, USDT/TON/BTC, без KYC для тебя, нативная кнопка Pay |
| **Cryptomus / Heleket** | Крипто-мерчант-шлюзы | ~0.4–1% | мерчант-аккаунт | — | Больше монет, вебхуки, низкая комиссия, трансгранично |

### Важный нюанс по YooKassa (два режима интеграции)
1. **Нативные Telegram Payments** — через `provider_token`, который выдаёт @BotFather после привязки. Проще всего, но **нельзя настроить рекуррентные платежи и холды**.
2. **Прямой YooKassa API + вебхук** — поддерживает автоплатежи/рекуррент, чеки 54-ФЗ, СБП, SberPay. Именно так делают серьёзные боты. Чуть больше кода, но для подписочной модели это то, что нужно.

### Рекомендуемое комбо для RU/СНГ
**YooKassa (карты/СБП, если есть ИП/ООО) + CryptoBot или Cryptomus (крипта, низкая комиссия и анонимность) + Telegram Stars (для iOS и анонимных).** Именно такой набор (или его подмножество) уже «из коробки» в `snoups/3xui-shop` и `remnashop` — ещё один аргумент за форк.

> ⚠️ Отдельно проверь юридическую сторону приёма платежей за VPN в твоей юрисдикции и требования 54-ФЗ (онлайн-чеки) при работе через RU-эквайринг. Это не юридическая консультация — уточни у профильного специалиста до запуска.

## 6. Рекомендуемая архитектура

Компоненты, независимо от пути B или C:

- **Слой Telegram-бота** (aiogram 3): меню, покупка тарифа, статус подписки, ссылки на конфиги/подписку, поддержка, реферальные ссылки.
- **Слой платежей**: провайдеры за единым интерфейсом (легко добавлять новые), вебхук-эндпоинты для колбэков (YooKassa/Cryptomus/YooMoney), обработчики `pre_checkout_query`/`successful_payment` для Stars и нативных TG-платежей.
- **Слой панели**: `py3xui` (3x-ui) или официальный SDK (Remnawave) — создание/продление/пауза клиента, лимиты по устройствам и трафику, срок действия, генерация subscription-ссылки.
- **Биллинг-ядро**: тарифы, подписки, история платежей, промокоды, рефералы; крон на истечение → напоминания → авто-отключение/авто-продление.
- **Админка**: управление серверами/пулом, статистика, редактор пользователей и тарифов, рассылки, бэкапы БД, режим обслуживания.
- **Инфра**: Docker Compose, reverse proxy + TLS (нужен валидный домен и сертификат под вебхуки и subscription-порт панели).

## 7. План действий (предлагаемый)

1. **Зафиксировать панель.** Решить: остаёмся на 3x-ui или переезжаем на Remnawave (рекомендация — Remnawave, если это растущий сервис; официальный `remnawave/migrate` для переноса).
2. **Взять `snoups/3xui-shop` (или `remnashop` под Remnawave) как базу**, поднять в Docker, подключить одну платёжку (YooKassa или CryptoBot) и прогнать полный цикл: покупка → создание клиента в панели → выдача конфига → истечение → продление.
3. **Добавить остальные платёжные рельсы** и промо/рефералку по мере надобности.
4. **n8n** оставить только под нотификации/служебные крон-задачи, если он где-то реально удобнее.
5. Если готовый бот не покрывает сценарии — точечно форкать и дорабатывать, а не переписывать с нуля.
