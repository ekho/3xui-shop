# Релиз 3xui-shop-bot

Процедура выпуска релиза для форка бота. Git-корень — корень проекта
(remote `ekho/3xui-shop`), все git-команды выполняются из него без префиксов.
Версия проекта хранится в `pyproject.toml` (поле `version`, строка ~3).

## Что такое «релиз» здесь
**Механизм релиза = аннотированный git-тег `X.Y.Z`** (plain, без префикса `v`), а НЕ
объект GitHub Release. Форк объектов GitHub Release не создаёт: `gh release list`
покажет только унаследованные апстримные `v0.7.x`, а `gh release view 1.14.0` вернёт
«release not found». Поэтому `gh release ...` в этом процессе НЕ используем.

- Каждая фича = minor-бамп (…1.13 → 1.14 → 1.15).
- Работаем прямо в `main` — все релизные коммиты на main.

## Три обязательных правила
1. Проверить последний релиз (последний git-тег) перед стартом.
2. Обновить версию в `pyproject.toml` — в том же feat-коммите, ДО тега.
3. Обязательно написать описание (тело коммита + сообщение тега).

## Шаги

### 1. Узнать текущий релиз (по git-тегам, НЕ по `gh release`)
```bash
git tag -l --sort=-v:refname | grep '^1\.' | head
```
Взять последний тег, выбрать следующий номер (обычно minor-бамп). На 2026-07-08
последний релиз — **1.19.1**. Если непонятен тип бампа — уточнить у пользователя.

### 2. Бампнуть версию в pyproject.toml
Отредактировать `version = "X.Y.Z"` в начале `pyproject.toml`. Эта правка
**фолдится в сам feat-коммит** (как в 1.13.0/1.14.0), не отдельным коммитом.

### 3. Собрать описание
Посмотреть, что изменилось, и сформулировать notes:
```bash
git log <прошлый-тег>..HEAD --oneline
```
Релиз без описания не выпускать.

### 4. Закоммитить (с согласия пользователя)
Формат коммита: subject на английском `feat: <...>`, тело — на русском (что/зачем).
```bash
git add -A
git commit -m "feat: <english subject>" -m "<русское тело: что и зачем>"
```
Коммитить/пушить только по явной просьбе пользователя.

### 5. Поставить аннотированный тег на релизном коммите
Формат сообщения тега: `X.Y.Z — <краткое описание на русском>`.
```bash
git tag -a X.Y.Z -m "X.Y.Z — <краткое описание на русском>"
```
Тег ставится на коммит с уже обновлённой версией (см. подводный камень ниже).

### 6. Публикация
Отправляет релизный коммит и аннотированный тег разом:
```bash
git push origin main --follow-tags
```

## Важный подводный камень (реальный промах на 1.12.0, 2026-07-07)
Бамп версии должен попасть В ТОТ ЖЕ коммит, что и тег, — **до** создания тега.
Иначе тег укажет на коммит со старой версией, а форс-передвижка уже опубликованного
тега блокируется гардрейлом (деструктив). В релизе 1.12.0 тег `1.12.0` ушёл на коммит
`e51178b` (где в pyproject всё ещё было `1.0.0`), а бамп доехал отдельным коммитом
`0ae87a4` уже ПОСЛЕ тега. С 1.13.0/1.14.0 бамп фолдится прямо в feat-коммит — так верно.

Примечание: версия из `pyproject.toml` нигде в рантайме не читается (ни
`importlib.metadata`, ни `__version__`, ни Docker `LABEL`), поэтому рассинхрон внутри
тега технически некритичен — но правило соблюдаем ради чистоты истории релизов.
